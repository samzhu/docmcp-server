# 技術研究與教學文件工作流程

## 概述

這個 Skill 幫助用戶研究任何開發技術，並整理成可供學習的教學文件。
特別適合開發新手，會主動引導學習方向。

```
┌─────────┐   ┌─────────┐   ┌─────────┐   ┌─────────┐   ┌─────────┐   ┌─────────┐
│ Phase 0 │──▶│ Phase 1 │──▶│ Phase 2 │──▶│ Phase 3 │──▶│ Phase 4 │──▶│ Phase 5 │
│  日期   │   │  釐清   │   │  研究   │   │  決策   │   │  文件   │   │  引導   │
└─────────┘   └─────────┘   └─────────┘   └─────────┘   └─────────┘   └─────────┘
     │             │             │             │             │             │
     ▼             ▼             ▼             ▼             ▼             ▼
  取得當前      理解用戶       深度搜尋      報告發現      整理成        詢問後續
  日期          真正需求       最新資料      等待選擇      教學文件      學習需求
```

---

## 適用範圍

### 程式語言
Java, Kotlin, Swift, Rust, Go, Python, TypeScript, JavaScript, C#, Ruby 等

### 框架與函式庫
Spring Boot, SwiftUI, UIKit, React, Vue, Angular, Django, FastAPI, Axum 等

### 架構設計
- 微服務架構 (Microservices)
- 事件溯源 (Event Sourcing)
- CQRS (Command Query Responsibility Segregation)
- DDD (Domain-Driven Design)
- 六角架構 (Hexagonal Architecture)
- Clean Architecture
- Serverless

### DevOps & 雲端
Docker, Kubernetes, CI/CD, AWS, GCP, Azure, Terraform 等

### 其他
資料庫、訊息佇列、API 設計、測試策略、效能優化等

## Phase 0：取得當前日期

**每次研究開始前必須執行：**

```bash
date +%Y-%m-%d
```

**用途：**
- 搜尋時使用當前年份
- 文件標註正確日期
- 確保資料時效性

---

## Phase 1：釐清需求

### 1.1 評估用戶的問題

用戶可能：
- 清楚知道要什麼 → 直接進入 Phase 2
- 只有模糊概念 → 需要協助釐清
- 完全是新手 → 需要更多引導

### 1.2 如果用戶講不清楚

**步驟：**

1. **先快速搜尋**，了解相關主題
   ```
   WebSearch: [用戶提到的關鍵字] overview types categories
   ```

2. **列出可能的方向**，詢問用戶：
   ```markdown
   我搜尋了一下「[關鍵字]」，找到幾個相關方向：

   1. **[方向 A]** - 簡述...
   2. **[方向 B]** - 簡述...
   3. **[方向 C]** - 簡述...

   你想研究哪個方向？或者有其他具體的問題？
   ```

3. **等待用戶確認**後再進入深度研究

### 1.3 確認研究範圍

在開始深度研究前，確認：
- 研究的具體主題
- 用戶的背景/程度（新手/有經驗）
- 特定的使用場景（如果有）

## Phase 2：深度研究

### 2.1 搜尋策略

**使用 WebSearch 進行多角度搜尋：**

```
搜尋 1: [主題] official documentation {當前年份}     ← 官方文件優先！
搜尋 2: [主題] best practices {當前年份}             ← 最佳實踐必找！
搜尋 3: [主題] tutorial {當前年份}
搜尋 4: [主題] vs [替代方案] comparison
搜尋 5: [主題] production tips / real world experience
```

**重要：優先使用英文關鍵字搜尋，英文資源通常更完整、更新。**

### 2.1.1 主動尋找最佳實踐

> ⚠️ **每次研究都必須包含最佳實踐！**

**搜尋關鍵字組合：**
```
[主題] best practices {當前年份}
[主題] design patterns
[主題] production ready
[主題] enterprise patterns
[主題] common mistakes to avoid
[主題] anti-patterns
[主題] real world examples
```

**最佳實踐來源優先順序：**

| 優先級 | 來源 | 說明 |
|--------|------|------|
| 🥇 1 | 官方最佳實踐指南 | Apple HIG, Google Style Guide, 官方 Best Practices |
| 🥈 2 | 大公司工程經驗 | Netflix, Uber, Airbnb 等如何在生產環境使用 |
| 🥉 3 | 知名技術書籍/作者 | Clean Code, Design Patterns, Martin Fowler 等 |
| 4️⃣ 4 | 社群整理 | Awesome lists 中的 best practices 章節 |

**文件中必須包含的最佳實踐內容：**
- ✅ 推薦的做法（Do's）
- ❌ 應避免的做法（Don'ts / Anti-patterns）
- 💡 實務經驗與技巧（Tips）
- ⚠️ 常見錯誤與陷阱（Common Mistakes）

### 2.2 資料來源優先順序

> ⚠️ **嚴格遵守以下優先順序，優先引用高優先級來源！**

```
┌─────────────────────────────────────────────────────────────┐
│                    資料來源優先順序                           │
├─────────────────────────────────────────────────────────────┤
│  🥇 第一優先：官方來源                                        │
│     └─ 官方文件、官方教學、官方部落格、官方 GitHub             │
│                                                             │
│  🥈 第二優先：大公司技術部落格                                 │
│     └─ Google, Netflix, Uber, Airbnb, Meta 等工程部落格      │
│                                                             │
│  🥉 第三優先：英文技術文章                                    │
│     └─ 知名作者、技術媒體、高品質英文部落格                    │
│                                                             │
│  4️⃣ 第四優先：社群資源                                       │
│     └─ Awesome lists, GitHub trending, 高票 Stack Overflow  │
│                                                             │
│  5️⃣ 第五優先：中文資源                                       │
│     └─ 確認資訊正確性後才引用，標註原始英文來源               │
└─────────────────────────────────────────────────────────────┘
```

**詳細說明：**

| 優先級 | 類型 | 來源範例 | 可信度 |
|--------|------|----------|--------|
| 🥇 1 | **官方文件** | Apple Developer Docs, Spring.io, Rust Book, Go.dev, docs.microsoft.com | ⭐⭐⭐⭐⭐ |
| 🥇 1 | **官方部落格** | Swift Blog, Spring Blog, Kotlin Blog | ⭐⭐⭐⭐⭐ |
| 🥇 1 | **官方 GitHub** | apple/swift, spring-projects/*, rust-lang/* | ⭐⭐⭐⭐⭐ |
| 🥈 2 | **大公司工程部落格** | Netflix Tech Blog, Uber Engineering, Airbnb Tech, Google AI Blog, Meta Engineering | ⭐⭐⭐⭐ |
| 🥈 2 | **雲端廠商文件** | AWS Docs, GCP Docs, Azure Docs | ⭐⭐⭐⭐ |
| 🥉 3 | **知名技術作者（英文）** | Martin Fowler, Uncle Bob, Kent Beck, ThoughtWorks | ⭐⭐⭐⭐ |
| 🥉 3 | **英文技術媒體** | InfoQ, DZone, Dev.to (高品質文章) | ⭐⭐⭐ |
| 4️⃣ 4 | **社群精選** | Awesome lists, GitHub trending | ⭐⭐⭐ |
| 4️⃣ 4 | **Stack Overflow** | 高票答案（100+ votes）| ⭐⭐⭐ |
| 5️⃣ 5 | **中文資源** | 需交叉驗證，標註原始來源 | ⭐⭐ |

### 2.3 搜尋語言原則

```
優先順序：英文 > 中文

原因：
1. 官方文件通常是英文
2. 英文資源更新更快
3. 技術術語原文更精確
4. 大公司部落格多為英文

中文資源使用時機：
- 用戶明確要求中文
- 作為補充說明
- 需標註對應的英文原始來源
```

### 2.4 收集的資訊

- 核心概念與原理
- 優缺點分析
- 使用場景
- 入門步驟
- 常見陷阱
- 學習資源
- **所有來源的 URL**（重要！）

## Phase 4：整理教學文件

### 4.1 文件位置與命名

```
docs/
├── INDEX.md                           ← 文件索引（必須更新！）
├── [category]/                        ← 分類目錄
│   └── [topic-name].md               ← 教學文件
```

**命名規則：**
- 使用小寫英文
- 單字間用連字號 `-`
- 例如：`event-sourcing.md`, `spring-boot-basics.md`

**分類流程：**

> ⚠️ **每個專案的分類都不同，必須先查看 INDEX.md！**

```
1. 讀取 docs/INDEX.md 的「分類說明」章節
2. 判斷文件應歸入哪個現有分類
3. 如果沒有適合的分類 → 詢問用戶是否新增
4. 新增分類後要同步更新 INDEX.md 的分類說明
```

**執行步驟：**

```bash
# Step 1: 先讀取 INDEX.md 了解現有分類
Read docs/INDEX.md
```

**判斷邏輯：**

```
文件類型判斷：
├── 專案專用文件（架構設計、設計語言等）→ project/
├── 通用教學文件 → 依主題分類（ios/, architecture/, tools/ 等）
└── 找不到適合分類 → 詢問用戶：
    「這份文件關於 [主題]，現有分類沒有完全符合的。
     建議新增分類 [建議名稱]，你覺得如何？」
```

**新增分類時：**
1. 建立目錄：`mkdir docs/[new-category]/`
2. 更新 INDEX.md 的「分類說明」表格
3. 在 INDEX.md 新增該分類的章節

## Phase 5：引導後續學習

### 5.1 主動詢問

文件完成後，主動詢問用戶：

```markdown
---

1. **[相關主題 A]** - [為什麼相關]

3. **[相關主題 C]** - [為什麼相關]

（如果你不確定從哪裡開始，可以告訴我你想做什麼專案，我幫你規劃學習路線）
```

### 5.2 針對新手的引導

如果用戶是新手，可以主動：
- 解釋專有名詞
- 建議學習順序
- 提供入門資源
- 詢問具體想做的專案，反推需要學什麼

### 5.3 引導問題範例

```markdown
為了更好地幫助你，可以告訴我：
```

## 注意事項

### 資料時效性
- 開始前先執行 `date +%Y-%m-%d`
- 搜尋使用當前年份
- 文件標註實際更新日期

### 來源標註（重要！）

**嚴格遵守優先順序：**
```
🥇 官方 > 🥈 大公司 > 🥉 英文文章 > 4️⃣ 社群 > 5️⃣ 中文
```

- **所有資訊必須標註來源**
- **優先引用官方文件**，沒有官方才找大公司
- **優先使用英文來源**，更新更快、更準確
- 使用表格格式，標註來源類型
- 來源可供其他 AI 參考，協助用戶深入學習
- 中文資源需標註對應的英文原始來源

### 文件管理
- 新增文件後必須更新 `docs/INDEX.md`
- 優先更新現有文件，避免重複
- 保持分類結構一致

### 用戶引導
- 不要假設用戶知道下一步該做什麼
- 主動提供學習建議
- 用簡單的語言解釋複雜概念
